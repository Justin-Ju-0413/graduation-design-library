## 第一章：这个项目到底在做什么？

### 一句话版本

我在一块 FPGA 开发板上，用 RISC-V 处理器 + 自己设计的硬件加速器，跑通了一个完整的手写数字识别（LeNet-5），速度比纯软件快 5 倍。

### "翻译成人话"版本

想象你有一块乐高底板（FPGA 芯片），上面可以自由搭电路。

- **RISC-V 处理器** = 一个迷你 CPU，像乐高小人，会执行程序
- **CNN 加速器** = 一个专用的"计算小工厂"，专门做卷积运算（AI 推理的核心计算）
- **NICE 接口** = CPU 和小工厂之间的"传话通道"

传统做法是 CPU 一条一条指令慢慢算。我们的做法是：CPU 遇到卷积计算时，把活丢给小工厂，"你给我算，算完叫我"，然后 CPU 继续干别的事。

**结果**：原来需要 1516 个时钟周期，现在只需要 287 个。快了 5.28 倍。识别准确率只掉了 1.7%（98.34% vs 100%）。

## 第二章：核心概念速成班

### 2.1 FPGA 是什么？

FPGA = 可编程逻辑门阵列。普通芯片（比如你手机的 CPU）电路是出厂就刻死的。FPGA 的电路可以随时重写——就像 Word 文档可以反复修改，FPGA 的"电路图"也可以反复下载。

### 2.2 RISC-V 是什么？

一种开源免费的 CPU 指令集。x86（Intel/AMD）和 ARM（手机芯片）都要收钱，RISC-V 不要钱，任何人都能用。所以大学和研究机构很喜欢用它。

我们这个项目用的是**蜂鸟 E203**，一个国产开源 RISC-V 核心。

### 2.3 CNN 加速器加速了什么？

CNN（卷积神经网络）的计算 90% 都是同一种操作：**乘加**。

```
输入像素 × 权重 → 加起来 → 存起来 → 下一个
```

一个卷积层可能有几千次乘加。CPU 一次只能做一个乘法，我们的硬件加速器一次能做 16 个（4×4 阵列同时算）。

### 2.4 NICE 是什么？

NICE = Nuclei Instruction Co-unit Extension（核芯指令协处理扩展）。

说白了就是 CPU 的"外挂接口"。CPU 执行到特殊指令（比如 `WLOAD` 加载权重），通过 NICE 接口把活派给加速器。加速器干完了，通过 NICE 接口告诉 CPU "我好了"。

### 2.5 ILA 是什么？

ILA = Integrated Logic Analyzer（集成逻辑分析仪）。

FPGA 内部的信号你是看不见的。ILA 就像给芯片内部装了一个"摄像头"，可以抓拍某个时刻所有信号的值。调试时极度重要——没有它，芯片内部就是黑盒子。

### 2.6 关键指标速查

| 指标 | 数值 | 含义 |
|------|------|------|
| 时钟频率 | 16 MHz | 芯片每秒跳 1600 万次 |
| PE 阵列 | 4×4 | 一次算 16 个乘加 |
| 数据精度 | INT8 | 权重和输入都用 8 位整数 |
| 加速比 | 5.28x | 287 cycles vs 1516 cycles |
| 准确率 | 98.34% | FPGA 上跑 10000 张图的结果 |
| 资源占用 | LUT 20.8%, FF 10.1%, BRAM 26.3% | 芯片还剩大量空间 |

## 第三章：完整开发时间线

### 第一阶段：纸上谈兵（2026 年 2-3 月）

**做了什么**：确定方案、写文档、画架构图。

这时候的想法：
- 用 NICE 接口连接 CPU 和加速器
- 4×4 的计算阵列
- 6 条自定义指令：CLEAR（清零）、WLOAD（加载权重）、DLOAD（加载数据）、COMP（开始计算）、RSTAT（读结果）

**通俗理解**：还没动工，先把设计图画好，把"我们想要什么"写清楚。

### 第二阶段：让仿真跑起来（2026 年 3 月 17-21 日）

**做了什么**：在电脑上用 iverilog 软件仿真了整个芯片。

仿真 = 在电脑上模拟芯片的行为。不需要真的 FPGA 板子，纯软件就能验证"电路逻辑对不对"。

**踩的第一个坑**：NICE 指令编码搞错了。

标准的 RISC-V 指令用 `funct3` 字段区分指令类型。但 E203 的 NICE 接口把 `funct3` 的位重新定义成了控制标志 `{xd, xs1, xs2}`（是否写目标寄存器、是否读源寄存器1、是否读源寄存器2）。

**通俗理解**：我们按照"标准英语"写指令，但 E203 的 NICE 接口说的是"方言"。同一个位置（比特位），它理解的意思不一样。

**解决方法**：改用 `funct7` 字段来区分指令。标准 RISC-V 里 `funct7` 是给复杂指令用的，但在 NICE 自定义指令里它是空的，正好拿来用。

**3 月 21 日**，仿真跑通了！软件程序能通过 NICE 接口正确地控制加速器，计算结果正确（RSTAT=320）。

**通俗理解**：在电脑模拟里，CPU 能正确地把活派给加速器，加速器也算对了。纸上方案验证通过。

### 第三阶段：准备上真板子（2026 年 3 月 21 日 - 4 月 10 日）

**做了什么**：
- 把仿真通过的代码准备好，准备烧到真 FPGA 上
- 搭建 Windows 下的 Vivado 编译环境
- 写了一个 PowerShell 脚本，一键编译 FPGA 固件

**关键决策：放弃 GDB 调试，改用 UART + LED + ILA**

原本计划用 GDB（就像调试普通程序一样，可以单步、打断点）来调试 FPGA 上的 CPU。但发现板子上的 JTAG 芯片不支持 CPU 调试。

**通俗理解**：本来想用"远程桌面"连到芯片内部一步步看，结果网卡不支持。只能改用"看日志 + 看指示灯"的方式。

**决策**：走"路线 A"——串口打印 + LED 灯 + ILA 抓信号。

这个决策后来证明是救命的关键。如果没有及时转向，板子调试会完全卡死。

### 第四阶段：板子调试——地狱级难度（2026 年 4 月 23-30 日）

这是整个项目最艰难的阶段。板子第一次上电，**什么输出都没有**。串口没有打印，LED 不亮，ILA 也连不上。

#### 4.1 从最简单的东西开始验证

面对一块"石头"一样的板子，我们采取了**逐层隔离**策略：

**第 1 步：验证时钟**
- 做了一个最简单的设计：计数器 + LED
- 结果：LED 闪烁了！说明芯片是活的，时钟在工作。

**第 2 步：验证 ILA**
- 用最原始的时钟（sys_clk）驱动 ILA
- 用 MMCM（时钟管理器）生成的 16MHz 驱动设计
- 结果：ILA 抓到数据！但发现一个奇怪的问题——如果用 MMCM 的时钟驱动 ILA，数据会损坏。

**发现的坑**：在这个 Vivado 版本 + 这个板子的组合下，MMCM 输出的时钟驱动 ILA 会导致数据上传损坏。解决办法是用原始 sys_clk 驱动 ILA（让 ILA 用独立时钟，不和设计共用一个时钟域）。

**通俗理解**：就像摄像头和被拍的东西要用不同的电源，否则画面会花。

**第 3步：验证 CPU**
- 烧入完整的 SoC 设计
- ILA 抓 PC（程序计数器）信号
- 结果：PC 在 0x00000000 和 0x00000002 之间跳——CPU 在启动 ROM 里踏步，没有跳到用户程序。

**通俗理解**：CPU 醒了，但在"开机画面"卡住了，没进系统。

#### 4.2 四个 Root Cause，缺一不可

经过几天排查，发现 CPU 起不来是**四个独立问题叠加**的结果：

| # | 问题 | 通俗解释 | 怎么修的 |
|---|------|---------|---------|
| 1 | ITCM 内存初始化格式错误 | CPU 的"课本"（程序存储器）内容格式不对，CPU 读出来是乱码 | 写了一个转换脚本，把编译好的程序转成存储器能认识的格式 |
| 2 | DTCM 内存初始化格式错误 | CPU 的"草稿纸"（数据存储器）同理 | 同上 |
| 3 | 宏定义没生效 | Vivado 编译时漏掉了一个开关 `E203_FORCE_BOOTROM_BOOT`，导致 CPU 不知道该从 ROM 启动 | 在 Vivado 工程里全局加了这个开关 |
| 4 | 仿真用的 #1 延时 | 仿真代码里有一行 `#1` 延时，在真硬件上不适用 | 删掉 |

**关键教训**：这四个问题只要有一个没修好，CPU 就起不来。而且它们之间没有明显的因果关系，必须一个个排查。仿真是救命稻草——在仿真里可以秒级迭代，FPGA 编译一次要 10 分钟。

**4 月 30 日**，串口终于打印出了：

```
hello_e203: boot
hello_e203: uart ok
hello_e203: loop
```

整个实验室沸腾了。

**通俗理解**：CPU 成功启动了，初始化了串口，进入了主循环。这是板子调试的"破冰时刻"。

#### 4.3 CNN 加速器上板验证（2026 年 4 月 30 日 - 5 月 6 日）

CPU 能跑了，下一步是让加速器也跑起来。

**又踩了一个坑**：UART 波特率算错了。

波特率 = 串口通信速度。原本代码里写的是 `DLL=27`，但实际上对于 16MHz 时钟和 115200 波特率，正确的值是 `DLL=138`。

**通俗理解**：两个人说话，一个人语速设为"快"，另一个人期待"慢"，结果谁也听不懂谁。改过来之后，串口输出正常了。

**5 月 5-6 日**，完整 LeNet-5 网络在 FPGA 上跑通：
- 10 张测试图全部分类正确（10/10）
- 完整 MNIST 测试集（10000 张）准确率 98.34%
- 比纯 CPU 快 5.28 倍

**又踩了一个坑**：ReLU 激活函数的位置。

Conv2（第二个卷积层）有多个输入通道。原来每个通道单独做 ReLU（把负数清零），再求和。但这不对——应该先求和，再做 ReLU。否则会丢掉"负数相消"的机会。

**通俗理解**：先算总账再看盈亏（先加总再清零），而不是每笔账都只看盈利（先清零再加总）。修完后准确率提升明显。

## 第四章：踩过的坑（避坑指南）

### 坑 1：工具链格式不兼容

**现象**：CPU 在仿真和真板子上都卡在 PC=0

**原因**：RISC-V 工具链的 `objcopy -O verilog` 输出的是按字节排列的十六进制，但 BRAM 的 `$readmemh` 期望的是按字（32位或64位）排列的。

**举例**：
- objcopy 输出：`@00000000 13 01 00 00 ...`（每字节一个数）
- BRAM 期望：`@00000000 00000113 ...`（每 4 字节拼成一个 32 位数，小端序）

**修法**：写了一个 PowerShell 转换脚本 `Convert-VerilogHex.ps1`

### 坑 2：MMCM 时钟不能驱动 ILA

**现象**：用 MMCM 生成的 16MHz 驱动 ILA，数据上传总是损坏

**原因**：这个 Vivado 版本 + 板子组合下，MMCM 输出时钟与 JTAG 时钟域不兼容

**修法**：ILA 用原始 sys_clk（独立时钟），设计用 MMCM 时钟。两者通过 CDC（跨时钟域）安全传递信号。

### 坑 3：Vivado 宏定义作用域

**现象**：Verilog 代码里的 `` `ifdef E203_FORCE_BOOTROM_BOOT `` 在 Vivado 里不生效

**原因**：这个宏定义在 `e203_defines.v` 里，但引用它的文件没有 `include` 这个头文件。仿真器会自动传播宏，Vivado 不会。

**修法**：在 Vivado Tcl 脚本里用 `set_property verilog_define {E203_FORCE_BOOTROM_BOOT=1}` 全局定义。

### 坑 4：UART 波特率算错

**现象**：串口输出乱码

**原因**：16MHz 时钟 / (16 × 115200) = 8.68，但实际需要的分频系数不是这么算的。UART IP 的分频公式不同，正确的 DLL 值是 138。

**修法**：查 UART IP 的数据手册，找到正确的分频公式。

### 坑 5：ReLU 放错位置

**现象**：Conv2 多通道卷积结果不对

**原因**：先逐通道 ReLU 再求和 ≠ 先求和再 ReLU

**修法**：把 ReLU 移到通道求和之后。

### 坑 6：NICE 指令编码用错字段

**现象**：WLOAD/DLOAD 指令不生效

**原因**：用了 `funct3` 但 E203 NICE 把它重定义为控制标志

**修法**：改用 `funct7` 区分指令。

## 第五章：关键架构决策

### 决策 1：为什么用 NICE 而不是内存映射？

| 方案 | 优点 | 缺点 |
|------|------|------|
| NICE（指令接口） | 延迟低，一条指令触发 | 最多 2 个源寄存器，数据量受限 |
| 内存映射 | 可以传大量数据 | 需要总线仲裁，延迟高 |

选了 NICE，因为卷积的权重和输入数据可以通过多条 WLOAD/DLOAD 分批传，而控制指令（CLEAR/COMP/RSTAT）需要低延迟。

### 决策 2：为什么是 4×4 阵列？

- 标准卷积核大小 3×3 到 5×5
- 4×4 可以覆盖大多数情况
- INT8 精度下，16 个乘加可以在一个周期完成
- 资源占用合理（不超 30%）

### 决策 3：为什么用 INT8 而不是 FP32？

- INT8 推理精度损失通常在 1-3%，我们实测 1.7%
- INT8 硬件面积约为 FP32 的 1/4
- FPGA 上的 DSP 单元原生支持 25×18 乘法，刚好够 INT8×INT8
- 对于嵌入式推理场景，INT8 是工业界主流选择

### 决策 4：为什么放弃 GDB 改用 ILA？

- 板子的 JTAG 芯片不支持 CPU 调试
- ILA 虽然不能单步，但可以抓 1024 个采样点的完整信号波形
- 配合 LED 和 UART，足够定位问题
- 如果坚持 GDB，项目可能在板子调试阶段就卡死了

## 第七章：如果你要接着做

### 可以改进的方向

1. **重新抓 ILA 数据**：目前的 ILA 没抓到 NICE 指令执行窗口。改触发条件，让 ILA 在 NICE CSR 变化时触发，就能抓到完整的 NICE 指令序列。

2. **性能优化**：目前 16MHz 下加速比 5.28x。提高时钟频率、增加 PE 阵列大小（8×8？）可以进一步提升。

3. **支持更多网络**：当前只验证了 LeNet-5。验证 ResNet、MobileNet 等更大网络需要修改软件驱动层。

4. **功耗测量**：论文里没有功耗数据。可以加一个功耗估算。

5. **与 CPU 的对比更精细**：目前的加速比只针对一个 3×3 卷积。对于完整 LeNet-5 推理（包含 FC 层、池化等），端到端加速比是多少？

### 常规操作流程

```bash
# 修改加速器代码
cd "New project/riscv_cnn_accelerator"
# 编辑 hw/rtl/acc/*.v

# 仿真验证
make sim          # RTL 单元测试
make sim_debug    # 带波形

# FPGA 编译（Windows PowerShell）
cd "New project/e203_hbirdv2"
powershell -File fpga/davinci_a7_100t/script/Invoke-Vivado-Fpga.ps1 -BuildMode cnn_sysclk_ila -Action bit

# 修改论文
cd "../Graduation_Design_Library/thesis_latex"
# 编辑 chapters/*.tex
bash build.sh all
```

### 最重要的原则

1. **先仿真，后上板**：仿真 10 秒，编译 10 分钟。先确认仿真过再烧 FPGA。
2. **一次只改一个东西**：多个改动叠加 = 不知道哪个起作用了。
3. **保留证据**：每次上板抓的 ILA CSV、串口截图都要存档。
4. **单信源原则**：论文内容只改 .tex，参考文献只改 Zotero，图片只改 .py 脚本。
